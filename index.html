How a single event flows end-to-end

1

Event generated

producer.py creates a JSON event with a unique event_id, a random event_type, and a nested payload dict containing IP, country, amount, and browser string.

2

Published to Kafka

The event is serialized to JSON and produced to the events topic. The user_id is used as the message key — Kafka hashes it to assign the message to one of the 3 partitions consistently.

3

Consumed from Kafka

consumer.py polls Kafka every second. Each message is decoded from JSON and annotated with kafka_offset and kafka_partition before being added to an in-memory buffer.

4

Appended to Snowpipe channel

When the buffer reaches 100 rows or 5 seconds have passed, each row is appended to its partition's Snowpipe Streaming channel via append_row(). The Kafka offset is passed as the offset_token for exactly-once deduplication.

5

Snowflake flush confirmed

The consumer calls wait_for_flush(), blocking until the SDK confirms all rows in all channels have been durably written to Snowflake cloud storage.

6

Kafka offsets committed

Only after the Snowflake flush is confirmed does the consumer commit the Kafka offsets. This guarantees that if the consumer crashes, the unconfirmed messages will be re-delivered and re-processed.

7

Queryable in Snowflake

Rows are immediately visible in KAFKA_DEMO.STREAMING.CLICKSTREAM. The payload column is a native VARIANT object — fields like payload:country and payload:amount can be queried directly with Snowflake's semi-structured syntax.

Delivery guarantees & design decisions

🔁

At-least-once deliveryKafka offsets are committed only after wait_for_flush() succeeds. A crash before commit causes re-delivery; the SDK deduplicates via offset tokens.

📌

Resumable channelsChannels are named partition_0, partition_1, etc. If the consumer restarts, it reopens the same channels and continues from the last committed offset.

🔐

PAT authenticationThe Snowpipe Streaming SDK authenticates using a Programmatic Access Token (PAT) scoped to the SYSADMIN role. No RSA keys required.

⚡

Serverless ingestionSnowpipe Streaming uses no Snowflake warehouse. Ingestion compute is managed by Snowflake internally and billed per row/GB rather than per second of compute.

How consumer.py works

consumer.py — core poll loop
# One channel per Kafka partition — opened lazily on first message
def get_channel(partition_id):
    if partition_id not in channels:
        ch, _ = sf_client.open_channel(channel_name=f"partition_{partition_id}")
        channels[partition_id] = ch
    return channels[partition_id]

# Poll Kafka — buffer rows by partition
while running:
    msg = kafka.poll(1.0)
    if msg:
        row = json.loads(msg.value())
        row["kafka_offset"]    = msg.offset()   # for traceability
        row["kafka_partition"] = msg.partition() # for traceability
        batch.setdefault(msg.partition(), []).append((row, str(msg.offset())))
        offsets_to_commit[msg.partition()] = msg.offset() + 1

    # Flush when buffer is full OR time interval has elapsed
    if total_buffered >= BATCH_SIZE or elapsed >= FLUSH_INTERVAL_SECS:
        for p, rows in batch.items():
            ch = get_channel(p)
            for row, offset_token in rows:
                ch.append_row(row, offset_token=offset_token)

        sf_client.wait_for_flush(timeout_seconds=30)  # blocks until SF confirms

        for p, offset in offsets_to_commit.items():
            kafka.commit(offsets=[TopicPartition(TOPIC, p, offset)])  # AFTER flush

Why manual Kafka offset commits?

Auto-commit advances the offset on a timer regardless of whether Snowflake has confirmed the write. If the consumer crashes between auto-commit and a successful flush, those messages are silently lost.

Manual commit after wait_for_flush() guarantees the Kafka offset only advances when data is durably in Snowflake.

Why one channel per partition?

Snowpipe Streaming channels are ordered sequences. Mapping one channel to one Kafka partition preserves per-partition ordering and keeps channel recovery simple — when a consumer instance picks up a partition after a rebalance, it just reopens the same named channel and continues.

Mixing partitions into a single channel would require coordination across threads.

Key design considerations

Delivery

Exactly-once via offset_token

Kafka delivers messages at-least-once — if the consumer crashes before committing an offset, those messages will be re-delivered on restart. The SDK's offset_token deduplication means duplicate deliveries are silently discarded at the Snowflake layer, giving you effectively exactly-once writes.

How this demo implements it: every append_row() call passes the Kafka offset as the offset_token. Kafka offsets are committed only after wait_for_flush() confirms the write. On restart, Kafka re-delivers any uncommitted messages — the SDK recognises the duplicate offset tokens and discards them automatically.

Tuning

Batch size vs. latency

BATCH_SIZE=100 and FLUSH_INTERVAL_SECS=5 are demo values. In production, higher batch sizes (1,000–10,000) reduce per-row overhead. Lower flush intervals reduce end-to-end latency. Tune based on your throughput and acceptable lag.

Failure

wait_for_flush() timeout

If Snowflake is under pressure, wait_for_flush(timeout_seconds=30) can raise a timeout exception. In production, this should be caught, logged, and retried with exponential backoff — the SDK's internal buffer preserves the rows until the next successful flush.

HA

High availability via consumer groups

Run multiple instances of consumer.py with the same group.id. Kafka automatically distributes the 3 partitions across live instances. If one crashes, Kafka rebalances its partitions to surviving instances within seconds.

Schema

Manual schema management

Unlike the Kafka Connector or Openflow, the SDK does not auto-evolve the Snowflake table. You are responsible for keeping the table DDL in sync with your data.

What that means in practice:

  • Add a new field to your Kafka messages → ALTER TABLE ... ADD COLUMN before deploying the new consumer
  • Remove a field → old rows keep the column; consumer must stop writing it or write NULL
  • Rename a field → add the new column, backfill, remove the old one manually
  • Change a type → requires a new column + migration; the SDK has no type-coercion layer

In this demo, payload is VARIANT — nested fields can change freely without a DDL change. Only the top-level typed columns (event_id, event_type, etc.) require a schema migration if they change.

Auth

PAT scope & expiry

The PAT used here is scoped to SYSADMIN. In production, create a dedicated service role with only INSERT on the target table — not SYSADMIN. PATs expire (default 15 days, up to 365 days) — build rotation into your deployment pipeline.

Running in production

🏗️ Deployment

  • 🐳
    ContainerizeBuild a Docker image from consumer.py + requirements.txt. Deploy as a Kubernetes Deployment or ECS task with replicas matching your partition count.
  • 📦
    Scale = partitionsSet replicas equal to the number of Kafka topic partitions. Each instance will own 1+ partitions via consumer group rebalancing.
  • 🔄
    Graceful shutdownSIGTERM is already handled — the consumer drains and flushes before exiting. Set terminationGracePeriodSeconds: 60 in your pod spec.
  • 📍
    Co-locate with KafkaRun consumers in the same cloud/region as your Kafka brokers to minimize read latency. The Snowflake write goes over HTTPS and is WAN-tolerant.

🔐 Security

  • 🔑
    Dedicated service roleCreate a role with only INSERT ON TABLE and USAGE ON DATABASE/SCHEMA. Never use SYSADMIN in production.
  • 🗝️
    PAT rotationGenerate PATs with a 30–90 day expiry. Store in a secrets manager (AWS Secrets Manager, HashiCorp Vault). Rotate before expiry and redeploy.
  • 🌐
    Network policyBind the Snowflake service user to a network policy that allows only the egress IPs of your consumer deployment. Block all other sources.
  • 🚫
    No secrets in codeRead credentials from environment variables or a secrets manager injection — never hardcode or commit them.

📊 Monitoring

  • 📈
    Prometheus metricsSet SS_ENABLE_METRICS=true to expose a built-in Prometheus endpoint on port 50000. Scrape channel throughput, flush latency, and error counts.
  • 🔍
    Channel health in SnowflakeQuery SHOW CHANNELS IN PIPE KAFKA_DEMO.STREAMING."CLICKSTREAM-STREAMING" to see per-channel status and last committed offset.
  • ⏱️
    Consumer lagMonitor Kafka consumer group lag separately (e.g., Kafka UI, Grafana). High lag means the consumer isn't keeping up with the producer.

⚡ Tuning for high throughput

  • 📦
    Increase batch sizeRaise BATCH_SIZE to 1,000–10,000 for high-volume workloads. Fewer, larger flushes are more efficient than many small ones.
  • 🔀
    More partitionsAdd Kafka topic partitions and matching consumer instances to scale horizontally. Each partition = one independent channel = linear throughput scaling.
  • 🧵
    Async appendFor very high throughput, use a threaded model: one thread per partition calling append_row() in parallel, with a single coordinating thread calling wait_for_flush().
  • 📉
    Suppress SDK logsSet SS_LOG_LEVEL=error in production to eliminate verbose INFO output and reduce log storage costs.

PAT Authentication Setup

The consumer authenticates to Snowflake using a Programmatic Access Token (PAT). It reads the PAT from ~/.snowflake/connections.toml, or from the SNOWFLAKE_PAT environment variable if set.

How to generate a PAT In Snowsight: Admin → Security → Programmatic Access Tokens → Create new token. Scope it to your role (e.g., SYSADMIN). Copy the token value — it is only shown once.
~/.snowflake/connections.toml — add this entry 
[my_connection]
account = "YOUR-ORG-ACCOUNT"
user    = "YOUR_USERNAME"
password = "pat_your_programmatic_access_token_here"

The connection name (e.g., my_connection) does not matter — config.py scans all entries and matches on account + user. Alternatively, skip connections.toml entirely and set SNOWFLAKE_PAT=pat_... as an environment variable before running consumer.py.

import tomllib

# Suppress Snowpipe Streaming SDK INFO/WARN logs — only show errors
os.environ.setdefault("SS_LOG_LEVEL", "error")


def _load_pat_from_connections_toml():
    toml_path = os.path.expanduser("~/.snowflake/connections.toml")
    with open(toml_path, "rb") as f:
        conns = tomllib.load(f)
    for conn in conns.values():
        if (
            conn.get("account", "").upper() == "YOUR-ORG-ACCOUNT"   # <-- edit this
            and conn.get("user", "").upper() == "YOUR_USERNAME"      # <-- and this
            and conn.get("password")
        ):
            return conn["password"]
    raise RuntimeError("PAT not found in ~/.snowflake/connections.toml")

consumer.py

Reads messages from Kafka and writes them to Snowflake via the Snowpipe Streaming Python SDK. Opens one channel per Kafka partition. Buffers rows and flushes every 100 rows or 5 seconds, committing Kafka offsets only after Snowflake confirms the write.

demos/snowpipe-streaming-sdk/consumer.py 
import json
import os
import signal
import tempfile
import time
from datetime import datetime, timezone

from confluent_kafka import Consumer, TopicPartition

import config

# Controls the main poll loop — set to False by signal handlers to trigger graceful shutdown
running = True


def handle_signal(sig, frame):
    # Catches SIGINT (Ctrl+C) and SIGTERM (docker stop / kill).
    global running
    running = False


signal.signal(signal.SIGINT, handle_signal)
signal.signal(signal.SIGTERM, handle_signal)

# Flush to Snowflake after this many rows are buffered across all partitions
BATCH_SIZE = 100

# Also flush if this many seconds have passed with any buffered rows (low-volume safety net)
FLUSH_INTERVAL_SECS = 5.0


def build_profile_file():
    # Write auth config to a temp file so the PAT is never stored on disk permanently.
    profile = {
        "authorization_type": "PAT",
        "url": f"https://{config.SNOWFLAKE_ACCOUNT}.snowflakecomputing.com:443",
        "account": config.SNOWFLAKE_ACCOUNT,
        "role": config.SNOWFLAKE_ROLE,
        "personal_access_token": config.SNOWFLAKE_PAT,
    }
    path = os.path.join(tempfile.gettempdir(), "sf_snowpipe_profile.json")
    with open(path, "w") as f:
        json.dump(profile, f)
    return path


def main():
    from snowflake.ingest.streaming import StreamingIngestClient

    profile_path = build_profile_file()

    # Default pipe name convention: <TABLE>-STREAMING (auto-created on first open_channel)
    pipe_name = f"{config.SNOWFLAKE_TABLE}-STREAMING"

    print(f"Connecting to Snowflake account {config.SNOWFLAKE_ACCOUNT}")
    print(f"Target: {config.SNOWFLAKE_DATABASE}.{config.SNOWFLAKE_SCHEMA}.{config.SNOWFLAKE_TABLE}")
    print(f"Pipe: {pipe_name}")

    sf_client = StreamingIngestClient(
        client_name="kafka-demo-consumer",
        db_name=config.SNOWFLAKE_DATABASE,
        schema_name=config.SNOWFLAKE_SCHEMA,
        pipe_name=pipe_name,
        profile_json=profile_path,
    )

    # One channel per Kafka partition. Named channels are resumable — if this consumer
    # restarts and picks up the same partition, it reopens the same channel and continues.
    channels = {}

    def get_channel(partition_id):
        if partition_id not in channels:
            ch, _ = sf_client.open_channel(channel_name=f"partition_{partition_id}")
            channels[partition_id] = ch
            print(f"Opened channel for partition {partition_id}")
        return channels[partition_id]

    kafka = Consumer({
        "bootstrap.servers": config.KAFKA_BOOTSTRAP_SERVERS,
        "group.id": config.KAFKA_GROUP_ID,
        "auto.offset.reset": "earliest",
        "enable.auto.commit": False,   # manual commit — only after Snowflake flush
    })
    kafka.subscribe([config.KAFKA_TOPIC])
    print(f"Subscribed to Kafka topic '{config.KAFKA_TOPIC}' on {config.KAFKA_BOOTSTRAP_SERVERS}")
    print("Press Ctrl+C to stop.\n")

    batch = {}             # {partition_id: [(row_dict, offset_token), ...]}
    offsets_to_commit = {} # {partition_id: next_offset_to_commit}
    last_flush = time.time()
    total_flushed = 0

    try:
        while running:
            msg = kafka.poll(1.0)

            if msg is None:
                pass
            elif msg.error():
                print(f"Consumer error: {msg.error()}")
            else:
                p = msg.partition()
                row = json.loads(msg.value())
                row["kafka_offset"] = msg.offset()
                row["kafka_partition"] = p
                batch.setdefault(p, []).append((row, str(msg.offset())))
                offsets_to_commit[p] = msg.offset() + 1

            total_buffered = sum(len(v) for v in batch.values())
            elapsed = time.time() - last_flush

            if total_buffered >= BATCH_SIZE or (elapsed >= FLUSH_INTERVAL_SECS and total_buffered > 0):

                for p, rows in batch.items():
                    ch = get_channel(p)
                    for row, offset_token in rows:
                        ch.append_row(row, offset_token=offset_token)

                # Block until Snowflake confirms all rows are durably written
                sf_client.wait_for_flush(timeout_seconds=30)

                # Commit Kafka offsets AFTER Snowflake flush is confirmed
                for p, offset in offsets_to_commit.items():
                    kafka.commit(offsets=[TopicPartition(config.KAFKA_TOPIC, p, offset)])

                total_flushed += total_buffered
                ts = datetime.now(timezone.utc).strftime("%H:%M:%S")
                print(f"[{ts}] Flushed {total_buffered} rows to Snowflake | total rows: {total_flushed} | partitions: {list(batch.keys())}")
                batch.clear()
                offsets_to_commit.clear()
                last_flush = time.time()

    finally:
        for ch in channels.values():
            try:
                ch.close()
            except Exception:
                pass
        sf_client.close()
        kafka.close()
        if os.path.exists(profile_path):
            os.remove(profile_path)
        print(f"\nShutdown complete. Total rows flushed: {total_flushed}")


if __name__ == "__main__":
    main()

Generates synthetic clickstream events using Faker and publishes them to the Kafka events topic at ~10 events/sec. The user_id is used as the Kafka message key, ensuring consistent partition assignment per user.

import json
import time
import uuid
from datetime import datetime, timezone

from confluent_kafka import Producer
from faker import Faker

import config

fake = Faker()


def delivery_report(err, msg):
    # Called asynchronously after each produce() attempt. Only prints on failure.
    if err:
        print(f"Delivery failed: {err}")


def main():
    p = Producer({"bootstrap.servers": config.KAFKA_BOOTSTRAP_SERVERS})
    count = 0
    print(f"Producing events to topic '{config.KAFKA_TOPIC}' on {config.KAFKA_BOOTSTRAP_SERVERS}")
    print("Press Ctrl+C to stop.\n")
    try:
        while True:
            event = {
                "event_id": str(uuid.uuid4()),
                "event_type": fake.random_element(["page_view", "click", "purchase", "signup"]),
                "user_id": str(uuid.uuid4()),
                "payload": {
                    "ip": fake.ipv4(),
                    "country": fake.country_code(),
                    "amount": round(fake.pyfloat(min_value=1, max_value=500), 2),
                    "browser": fake.user_agent(),
                },
                "event_ts": datetime.now(timezone.utc).isoformat(),
            }

            # user_id as key ensures consistent partition assignment (hash-based)
            p.produce(
                config.KAFKA_TOPIC,
                key=event["user_id"],
                value=json.dumps(event),
                callback=delivery_report,
            )
            count += 1

            if count % 100 == 0:
                p.flush()
                print(f"Produced {count} events")

            time.sleep(0.1)  # ~10 events/second

    except KeyboardInterrupt:
        p.flush()
        print(f"\nStopped. Total produced: {count}")

How a file flows end-to-end

1

File lands in cloud storage

A .jsonl.gz file is uploaded to the external stage path (e.g., s3://your-bucket/kafka-demo/clickstream_batch_001.jsonl.gz).

2

Storage event notification fires

The cloud provider emits an event (S3 Event Notification → SQS, Azure Event Grid, or GCS Pub/Sub) indicating a new object was created.

3

Snowflake SQS queue receives event

Snowflake's internal SQS queue (configured during pipe creation) receives the notification and enqueues the file for loading.

4

Snowpipe loads file via COPY INTO

Snowpipe serverless compute executes the COPY INTO statement defined in the pipe, parsing the JSON and inserting rows into the target table. Compute is billed at 1.5x serverless credit rate.

5

Rows queryable in CLICKSTREAM

Rows are immediately visible in KAFKA_DEMO.BATCH.CLICKSTREAM. The file is recorded in Snowpipe's load history (64-day window) to prevent duplicate loading if the same file appears again.

AUTO_INGEST Mechanism

When AUTO_INGEST = TRUE, Snowpipe subscribes to cloud storage event notifications. The mechanism varies by cloud provider:

AWS

S3 Event → SQS

S3 bucket sends s3:ObjectCreated:* events to an SQS queue owned by Snowflake. The queue ARN is returned by SHOW PIPES — you configure S3 to publish to it.

Azure

Blob → Event Grid → Queue

Azure Blob Storage emits events via Event Grid. Snowflake creates a Storage Queue subscription. You authorize the integration via Azure AD consent.

GCP

GCS → Pub/Sub

GCS sends object finalize notifications to a Pub/Sub topic. Snowflake creates a subscription to that topic. You grant publish permissions to the GCS service account.

ALTER PIPE ... REFRESH is not for regular use Per Snowflake documentation, ALTER PIPE ... REFRESH is intended for one-time recovery or backfilling — not as a regular ingestion trigger. For ongoing ingestion, AUTO_INGEST with event notifications is the correct approach.

Deduplication & Best Practices

Snowpipe maintains a 64-day load history per pipe. If the same file (identified by path + size + last-modified) appears again within that window, it is silently skipped. This provides file-level exactly-once semantics without any application-side logic.

Best practices for files:

  • Compress files with gzip — Snowflake auto-detects gzip and decompresses on load
  • Use immutable file names (include a batch ID or timestamp) so file identity is stable
  • Aim for files between 100 MB – 250 MB (compressed) for optimal load performance
  • Avoid modifying files after upload — Snowpipe may not re-process modified files within the dedup window
Decoupling note File generation and file landing in cloud storage are decoupled steps. In production, the customer environment (e.g., Azure ADLS) determines how files arrive in the stage. The demo's sample files simply represent what upstream producer output looks like — the "how they get there" is environment-specific and out of scope for this demo.

1. Create Snowflake Objects

demos/stage-snowpipe/setup/snowflake_setup.sql 
USE ROLE SYSADMIN;

CREATE DATABASE IF NOT EXISTS KAFKA_DEMO;
CREATE SCHEMA IF NOT EXISTS KAFKA_DEMO.BATCH;

CREATE OR REPLACE TABLE KAFKA_DEMO.BATCH.CLICKSTREAM (
    event_id        VARCHAR,
    event_type      VARCHAR,
    user_id         VARCHAR,
    payload         VARIANT,
    event_ts        TIMESTAMP_NTZ,
    kafka_offset    NUMBER,
    kafka_partition NUMBER,
    ingest_file     VARCHAR
);

CREATE OR REPLACE PIPE KAFKA_DEMO.BATCH.CLICKSTREAM_PIPE
  AUTO_INGEST = TRUE
AS
COPY INTO KAFKA_DEMO.BATCH.CLICKSTREAM
    (event_id, event_type, user_id, payload, event_ts,
     kafka_offset, kafka_partition, ingest_file)
FROM (
    SELECT
        $1:event_id::VARCHAR,
        $1:event_type::VARCHAR,
        $1:user_id::VARCHAR,
        $1:payload,
        $1:event_ts::TIMESTAMP_NTZ,
        $1:kafka_offset::NUMBER,
        $1:kafka_partition::NUMBER,
        METADATA$FILENAME
    FROM @KAFKA_DEMO.BATCH.CLICKSTREAM_STAGE
)
FILE_FORMAT = (TYPE = 'JSON');
